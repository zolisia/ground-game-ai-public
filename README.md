# GroundGame Intel

A constituency intelligence dashboard for UK Members of Parliament and their staff. It aggregates open data — crime, planning, flood warnings, parliamentary activity, demographics, public health, schools, and more — for a single parliamentary constituency, surfacing what's happening on the ground in one place.

Currently piloted on **Braintree**. The architecture is constituency-scoped (lat/lng, ward codes, sample points, and a GeoJSON boundary) so other constituencies can be added by swapping the location data.

---

## What it shows

Data is grouped into themes on the dashboard. Each theme is backed by one or more API routes under [src/app/api/](src/app/api/).

| Theme | Sources |
|---|---|
| **Local issues** | UK Environment Agency flood warnings, police.uk crime reports, PlanIt planning applications, FixMyStreet community reports |
| **Parliamentary activity** | UK Parliament Members API (MP info, votes), Hansard (speeches, written questions), Parliament Petitions |
| **Demographics & profile** | ONS Census 2021 (ward-level choropleths), Commons Library constituency data |
| **Economy** | NOMIS / ONS labour market data (employment, unemployment, claimant count), HM Land Registry (UK House Price Index, Price Paid Data) |
| **Public services** | DfE GIAS (schools + Ofsted ratings), CQC (care home / GP inspection ratings), PHE Fingertips (public health indicators) |
| **Environment** | OpenAQ air quality, DLUHC EPC ratings, OpenStreetMap places of worship |
| **Media & politics** | Local news RSS, BBC / Politico headlines, Wikipedia + Electoral Calculus polling, X / Apify social mentions, SerpAPI Google Trends |
| **AI synthesis** | Daily intelligence brief generated by Anthropic Claude from the above sources |

## Architecture: cache-then-refresh

The single most important pattern in the codebase. Every API route under [src/app/api/](src/app/api/) follows the same shape:

1. **Read** the route's Firestore document on every request (~50–100 ms).
2. **Return cached data immediately** if it exists — the user never waits for the upstream API.
3. **In the background**, if the cache is older than the route's TTL, fetch fresh data from the upstream API and write it to Firestore — but only if the new data actually differs from what's cached (compared via `JSON.stringify`).
4. **On a cold cache** (first ever request, or after manual cache clear), do the upstream fetch synchronously, write to Firestore, and return the fresh data.

The result: dashboard widgets render in a fraction of a second after the first cache fill, even when the underlying upstream APIs take 10–60 seconds. Government APIs are slow; users don't have to know.

### Per-route TTLs

TTL is calibrated to how often the underlying data actually changes:

| Route | TTL |
|---|---|
| crime | 15 min |
| ai-brief | 30 min |
| fixmystreet | 30 min |
| planning | 1 hour |
| health, house-prices, cqc, employment, universal-credit, commons-library | 24 hours |
| schools, census | 7 days |
| floods | refresh on every request (small data, frequently relevant) |

### Firestore document layout

- Each route has its own collection: `flood_cache`, `crime_cache`, `planning_cache`, etc.
- Each collection has one document per constituency, keyed by name (currently `braintree`).
- Document shape: `{ data: <route-specific payload>, updated_at: ISO 8601 string }`
- Special case: `census_cache` uses one document per topic (e.g. `braintree-age-under16`, `braintree-ethnicity`) since the route response shape varies by `?topic=` parameter.

The Firestore client is initialised once in [src/lib/firebase.ts](src/lib/firebase.ts).

## Getting started

### Prerequisites

- Node.js 20+
- A Firebase project with Firestore enabled
- (Optional) Additional API keys to enable specific features — see [Environment variables](#environment-variables) below

### Setup

```bash
# 1. Clone and install
git clone <repo-url>
cd ground-game-ai-public
npm install

# 2. Create .env.local with the variables below

# 3. Run the dev server
npm run dev
```

Open [http://localhost:3000](http://localhost:3000) (or 3001 if 3000 is taken) and you'll see the dashboard. The first load will be slow as Firestore caches fill; subsequent loads will be near-instant.

### Build for production

```bash
npm run build
npm start
```

## Environment variables

Put these in `.env.local` (which is git-ignored). All `NEXT_PUBLIC_*` variables are exposed to the browser; non-public ones stay server-side.

### Required: Firebase

Get these from your Firebase project console (Project Settings → General → Your apps).

```
NEXT_PUBLIC_FIREBASE_API_KEY=
NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN=
NEXT_PUBLIC_FIREBASE_PROJECT_ID=
NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET=
NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID=
NEXT_PUBLIC_FIREBASE_APP_ID=
NEXT_PUBLIC_FIREBASE_MEASUREMENT_ID=
```

### Optional: feature-specific keys

Each unlocks one feature. Without them, the corresponding widget either falls back to static data or returns a placeholder. Nothing breaks.

| Variable | Enables |
|---|---|
| `ANTHROPIC_API_KEY` | Daily AI brief at `/api/ai-brief` |
| `EPC_API_KEY` + `EPC_EMAIL` | Energy Performance Certificate data at `/api/epc` |
| `APIFY_API_TOKEN` | X / Twitter mentions and opposition tracking |
| `SERPAPI_KEY` | Google Trends data |
| `X_BEARER_TOKEN` | Direct X API access (alternative to Apify) |

## Deployment

Designed to deploy on **Vercel**. The Firebase Firestore caching layer is provider-agnostic (works on any Node host), but the Next.js features used (`force-dynamic`, route handlers, `next/font`) are well-supported on Vercel out of the box.

### Environment configuration

Set the same environment variables (Firebase + any optional keys listed above) in your Vercel project settings before the first deploy. Vercel injects them at build time for `NEXT_PUBLIC_*` vars and at runtime for the rest.

### Branch → environment mapping

| Branch | Vercel environment | URL |
|---|---|---|
| `main` | Production | Project's primary domain |
| `zoja/dev` | Preview | Auto-generated preview URL per push |
| Any other branch | Preview | Auto-generated preview URL per push |

Production deploys are triggered automatically when a PR from `zoja/dev` is merged into `main`. Pushes to `zoja/dev` get their own preview URL, which is the recommended way to verify changes before opening the PR.

Each preview deploy gets its own URL but **shares the same Firestore project** as production by default. If you want preview deploys to use a separate cache, point `NEXT_PUBLIC_FIREBASE_PROJECT_ID` (and the related Firebase vars) at a second Firebase project for the Preview environment in Vercel's settings.

### Cache warm-up after deploy

Because every API route does a cache read on first request, the very first hit to a new deploy on a fresh Firestore database triggers synchronous upstream fetches (the cold-cache path described above). Expect the first dashboard load on a brand new environment to be slow; subsequent loads return cached data immediately while refreshes happen in the background.

## Adding a new constituency

The codebase is currently hardcoded to Braintree across several files. To support another constituency you would need to:

1. Update the lat/lng centre, radius, and bounding box in each `/api/*/route.ts` (or refactor these into a shared config).
2. Replace the GeoJSON boundary used by `isInsideConstituency` in [src/lib/geo.ts](src/lib/geo.ts).
3. Replace the ward codes in [src/app/api/census/route.ts](src/app/api/census/route.ts).
4. Replace the sample points in [src/app/api/crime/route.ts](src/app/api/crime/route.ts) and the postcodes in [src/app/api/cqc/route.ts](src/app/api/cqc/route.ts).
5. Update the constituency document key (`braintree`) used in every Firestore cache write.

Refactoring these into a single constituency config would be a sensible next step.
